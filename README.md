# Capability Memory - 万物基于指针的操作系统架构原型

> 一个探索“万物基于指针”操作系统的概念验证项目。通过软件模拟验证静态去重、动态版本控制、单地址空间等核心机制。

---

## 简介

这是一个**概念验证原型**，用于探索一个操作系统架构假设：让硬件直接理解“对象”和“指针”，所有数据在同一个地址空间，相同内容只存一份，修改产生新版本，由硬件自动管理内存。

本仓库实现了一个**软件模拟版本**，在 Linux 上用 `mmap` 分配单地址空间，用 C 库模拟核心机制。虽然性能不如理想硬件实现，但**证明了核心逻辑的可行性**。

---

## 核心设计

### 1. 能力指针

指针不再是物理地址，而是包含对象ID、版本号、权限的“能力”：

```c
typedef struct {
    uint64_t object_id;   // 对象唯一ID
    uint64_t version;     // 版本号
    uint8_t  permissions; // 读/写/执行权限
} capability_t;

---

### 2. 静态指针去重

相同内容的对象只存储一份，所有引用指向同一份数据。类似字符串驻留（String Interning），但推广到所有对象类型。

**核心机制**：
- 创建对象时计算内容哈希
- 查询全局哈希表
- 若存在相同内容，增加引用计数，返回已有指针
- 若不存在，分配新内存，写入数据，加入哈希表

**内存效率**：重复数据场景下，内存节省可达 50%-90%。

### 3. 动态指针版本管理

每次修改对象产生新版本，旧版本保留（写时复制）。类似 Git 的 commit 模型，天然支持快照和回滚。

**核心机制**：
- 修改时分配新内存
- 复制新内容到新内存
- 创建新版本元数据，版本号递增
- 旧版本通过版本链保留
- 返回新版本指针

**版本化指针结构**：

```c
typedef struct {
    uint64_t object_id;   // 对象唯一ID（不变）
    uint64_t version;     // 版本号（递增）
} versioned_ptr_t;

---
### 4. 内存地址Hash模糊匹配

用对象 ID 或内容哈希决定存放区域，相关对象的哈希值相近，物理位置相邻，提升 CPU 缓存命中率。

**核心机制**：
- 将地址空间划分为 1024 个区域，每个 1MB
- `region_id = hash(object_id) % 1024`
- 区域内顺序分配，保证连续性
- 无需维护映射表，哈希即地址

### 5. 指针最小调用路径

检测热路径，为高频访问的源-目标对创建直接捷径指针，减少多跳访问延迟。

**核心机制**：
- 记录指针访问轨迹
- 识别高频访问的源-目标对
- 在源对象中创建直接指向目标的捷径
- 捷径缓存满时 LRU 淘汰

### 6. 可达性回收机制

静态对象用引用计数回收，动态对象的版本链跟随当前版本回收，避免内存泄漏。

**核心机制**：
- 静态对象：引用计数归零时立即回收
- 动态对象：当前版本不可达时，整个版本链回收
- 循环引用：引入弱引用 + 周期可达性扫描

## 性能说明

当前版本是**软件模拟**，比直接内存访问慢约2倍（查表开销）。

但它的价值在于：
- **证明逻辑正确**：去重、版本控制、区域分配都能工作
- **展示内存节省**：重复数据场景节省可达90%以上
- **验证系统调用消除**：初始化后所有操作在用户态完成

如果硬件支持（区域标注、能力指令），性能预计可提升 **100倍以上**。
